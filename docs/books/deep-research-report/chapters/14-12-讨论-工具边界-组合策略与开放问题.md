# 12 讨论：工具边界、组合策略与开放问题

“Thinking with video”的一项核心诉求，是把原本被一次性压缩为静态上下文的视频输入，重新变成可被反复访问、可被验证的“证据工作台”。在这一语境下，“什么算工具”的边界往往比“工具能做什么”更难界定：同一项能力既可能以外部可调用的算子形式出现（如裁剪、取帧、检索与验证），也可能被吸收入端到端模型的内部策略（如隐式的采样预算分配、隐式注意力路由）。例如 Video-CoM明确将交互步骤的动作空间限定为三个原子操作（find-segment、find-frame、spatial-zoom），并把推理建模为由文本与操作交织形成的多轮轨迹 $\tau$ ；视频被预先等长分段，段/帧索引直接叠加在画面上，使得“调用哪个操作、参数是多少、得到什么新证据”都具有可审计接口 [5]。MedScope 进一步把这一接口规范化为由 <think>/<tool_call>/<tool_response>/<answer> 组成的结构化轨迹，并把工具空间收敛为两个“原生工具”：crop_video（输入时间窗，返回窗内密集采样帧序列以做局部证据获取）与 get_frame（输入时间戳，返回临近的三帧以做细粒度核验）[6]。在这种实现中，工具之所以被称为“工具”，并不只是因为它们执行了某种视觉运算，而是因为它们具有明确的参数化输入、可复现的可观测输出、并能作为推理链中的外显中间变量参与训练信号与评估。

与之相对，一些工作把“工具”扩展到更抽象的执行器与路由器：VideoExplorer 在框架层面把planner、temporal grounder 与 perception 解耦，尤其强调 temporal grounder 在离线阶段把视频分段并存储为多模态嵌入，在线阶段对文本或多模态查询进行匹配，并把子查询转化为构造（construc-tion）、检索（retrieval）、验证（verification）、总结（summarization）等动作再返回给 planner[3]。这里的“工具”不一定是单一算子，而是一个可编排的子系统：它既可以作为推理链的外部执行器降低主链复杂度，也可以引入验证器来过滤无关片段、缓解无证据的幻觉扩散 [3]。进一步地，ModaRoute把“是否调用某个索引/模态”也视为可由 LLM 驱动的工具路由问题：其路由器（GPT-4.1）在ASR/OCR/视觉三类索引之间做选择性查询，平均每条查询仅检索 1.78 个模态而非穷举 3.0 个模态，在维持 Recall $\textcircled { \mathrm { { a } } 5 { = } 6 0 . 9 \% }$ 的同时实现约 $4 1 \%$ 的计算开销降低 [7]。从边界讨论的角度看，这类“路由—检索—融合”的决策本身并不直接改变视频内容，但它决定了后续推理链可见的证据集合，因而在系统层面与裁剪、取帧同样具有“可调用、可编排、可度量成本”的工具属性。

工具边界的另一条分界线，是“外部工具”与“内部能力/提示策略”的混杂。VITAL这类工具增强强化学习框架，为了稳定思维与工具调用，会强制模型把交互轨迹结构化为<think>…</think><tool_call>{…}</tool_call>…<answer>…</answer> 并施加严格的规则格式奖励 [2]；MedScope 同样把结构化接口纳入训练与奖励的一部分（其复合奖励包含答案正确性、格式符合性与证据对齐项）[6]。这些做法在工程上常被归入“提示/格式约束”，但它们在训练闭环中扮演的角色更接近“软工具”：它们不直接提供视觉证据，却在约束行动空间、降低行为歧义、使得验证与再检索可实施方面起到关键作用。相反，某些看似“工具化”的能力（例如“密集采样 vs 粗采样”）也可能只是模型内部的策略变量。VideoExplorer 讨论了在不同任务中对指定时间区间进行密采样（细粒度）或粗采样（全局）的动态调整，以在上下文窗口受限下平衡可扩展性与细节保真 [3]；VideoTemp-o3 也把“裁剪后密集采样”视为长视频场景下替代均匀采样不足的关键手段 [4]。当这些策略被显式为可调用模块（如 crop_video 的返回密集帧序列 [6]）时，它们更容易被当作工具；当它们仅以内隐的采样参数出现时，则更像模型内能力。因而，“工具”边界的可操作定义往往需要回到接口层：是否存在可独立复现的输入输出协议、是否能被单独计费/计时、是否能在轨迹上被显式评价。

在工具链组合与编排上，现有 think-with-video 系统呈现出高度一致的“粗到细（coarse-to-fine）”范式，但各工作对“粗”“细”的落点不同：有的在时间上分层，有的在证据形态上分层，有的在

模态上分层。MedScope 明确借鉴临床医生“先全局 skim 再局部回看”的流程，把证据获取组织为粗到细的时间定位与反复核验，并在工具层面给出“时间窗裁剪 时间戳取帧”的层级检查通道 [6]。VideoTemp-o3 把 think-with-videos 直接形式化为 localize-crop-answer，并支持多轮grounding refinement：其数据构建流程中先由外部模型预测候选时间段并裁剪，再通过闭环一致性验证保留样本；对长视频（ $\mathrm { > 3 ~ m i n }$ ）才允许额外一轮 refinement，从而形成 multi-tool-call 训练数据 [4]。Video-CoM 则把粗到细落在“从段到帧到空间区域”的操作层级：先 find-segment 回看短clip，再 find-frame 冻结关键帧，最后 spatial-zoom 裁剪放大以读出细节 [5]。这类层级编排的共同点在于把高成本的高分辨率感知延后到“已知值得看”的局部区域，减少把长视频全部高密度编码带来的 token 与计算浪费；VideoExplorer 的 token 效率分析也体现了这一点：通过聚焦高相关片段实现“更少视觉 token 但更好性能”[3]。

“检索—验证—再检索”是另一类常见编排，它把工具链从“取证”扩展到“证据可靠性控制”。VideoExplorer 的 temporal grounder 在返回片段前引入 VLM verifier 来判断检索片段是否相关，不相关则丢弃，以避免无关片段进入主推理链造成幻觉 [3]。VideoTemp-o3 的闭环一致性验证（只保留答案与 ground-truth 一致的样本）把验证前移到数据构建阶段，通过训练数据筛选为多轮工具调用提供更稳定的监督 [4]；MedScope 在数据侧同样引入跨模型过滤与“multimodal confirmation”，要求直接从裁剪片段 ${ \mathrm { C l i p } } ( V , w )$ 回答，以排除 caption 幻觉与知识捷径，从而确保样本确实由局部视频证据支撑 [6]。这些设计暗示了一个重要组合原则：在长视频任务中，工具调用的收益并不只来自“看到更多”，而是来自“看到且能证明自己看到的是对的”。当验证环节缺失时，工具链可能把噪声证据更高分辨率地放大，从而加速推理漂移。

然而，工具链并非“越多越好”，而且错误往往以非线性的方式放大。VITAL 的工具消融显示：在时序定位任务中加入 clip caption 或 clip QA 工具不仅无益，甚至会导致 mIoU 出现显著下降；而video clipping 相较其他工具在经验上更能降低幻觉 [2]。这一结果提示工具链组合存在强任务依赖与强交互项：某些工具在独立评估中看似“提供信息”，但在端到端闭环中可能引入冗余与噪声、或改变模型的注意力分配方式，从而伤害定位精度。GLIMPSE 的分析从评测侧给出了相似信号：随着输入帧数增加，模型准确率先提升、超过某个阈值后收益递减甚至下降，作者将其归因于冗余信息与难以聚焦相关特征 [12]。这在工具视角下意味着：无论是“多看帧”还是“多调用工具”，都需要有路由与压缩机制来对抗冗余，而不能将信息量当作单调的性能杠杆。

泛化与迁移问题在 think-with-video 场景尤为突出，因为工具链把“任务解决方案”显式绑定在特定接口、索引与数据分布上。首先，跨时长的迁移仍未被充分覆盖：GLIMPSE 将视频长度限制在 20 秒到 2 分钟以满足其任务设计与标注边界 [12]，而 Video-VER 的局限性也指出其主要研究对象是“moderate length”的视频，并将非常长视频（尤其关键证据稀疏或时间模式复杂）作为未来方向 [10]。这意味着许多在中短视频上有效的工具编排（例如一次裁剪或一次检索）在超长视频上可能因证据稀疏而失效，需要更强的“多轮检索与证据记忆”。其次，跨领域迁移会暴露工具定义与误差模型的差异：MedScope 明确面向临床长视频，并强调其潜在收益与部署风险，指出不应作为独立临床使用、需要鲁棒性评估与人类监督 [6]。在此类高风险领域中，工具调用不仅要“找得到证据”，还要能够表达不确定性与失败模式；而许多通用领域的工具链（如基于 IoU 的定位奖励、基于选择题的自动评测）在医学领域可能带来不同的误判成本。

数据与可复现性构成了当前工具使用研究的系统性瓶颈。一方面，多步工具轨迹往往依赖强大的外部闭源模型生成或筛选：Video-CoM 用 Gemini-2.5-Pro 生成每个视频 3–5 个 QA（总计 25K），经验证后保留约 $7 5 \%$ 得到 18K，高质量数据中 15K 用于 SFT、3K 用于 GRPO，并且 RL 部分还额外策划 70 个 YouTube 多上下文视频 [5]；VideoTemp-o3 亦在数据构建中使用 Gemini-2.5-Pro 预测候选时间段与多轮 refinement，并用闭环一致性验证筛选样本 [4]；ModaRoute 的路由组件依赖GPT-4.1[7]；VideoExplorer 则指出由于缺乏高质量推理导向训练数据，借助强 LLM 结合其框架生

成推理数据，并以“仅保留正确答案轨迹”的方式确保有效性，最终整理 11.1k planner trajectories与 10.8k grounding trajectories[3]。这些做法在短期内提高了数据质量与系统能力，但也带来两类复现风险：其一是闭源模型与提示词不可控导致的数据分布难以复刻；其二是“只保留正确轨迹”的筛选策略可能引入系统性的选择偏差，使训练数据更接近外部教师模型的能力边界而非真实任务分布 [3]。另一方面，即便使用开源外部模型，数据生成也可能引入隐性偏差：Video-VER 在训练阶段用 Qwen2.5-VL-72B 生成问题相关视觉证据，但测试时策略模型独立运行、不依赖外部模型 [10]。这种“训练期工具、推理期不用”的设计有助于部署效率与接口简化，却要求研究者更清楚地报告证据生成器的配置、提示与失败率，否则难以判断性能提升来自证据对齐本身还是来自教师模型的偏置迁移。

鲁棒性风险往往来自工具链中的“中间决策点”，尤其是验证器与路由器的误判。VideoExplorer依赖 verifier 判断片段相关性并丢弃无关结果，这确实能缓解无关片段引发的幻觉，但同时也引入“证据遗漏”的新风险：一旦 verifier 误过滤关键片段，planner 将在缺证据情况下继续推理并可能产生高置信错误 [3]。ModaRoute 的误差分析揭示了另一种系统性脆弱性：OCR 查询的路由准确率仅$4 2 . 9 \%$ ，且主要错误来自字幕导致的 ASR/OCR 混淆、以及视觉描述与文本线索的重叠，使得路由器呈现“遗漏 OCR 的保守偏置”[7]。对视频推理而言，这类偏置意味着某些证据类型会在工具链入口处被系统性屏蔽，后续再复杂的推理与验证也无法弥补。与之相关，Video-VER 的失败案例指出“不完整的帧采样”会遗漏关键视觉上下文 [10]，而 GLIMPSE 的实验则显示“帧过多”也可能因冗余而削弱聚焦能力 [12]。两者共同指向一个鲁棒性悖论：采样过少会漏证据，采样过多会淹没证据；真正稳定的系统需要把采样本身变成可学习、可解释、可验证的工具策略，而不仅是固定超参数。

强化学习与奖励设计在工具使用研究中同时扮演“推动器”与“风险源”。VideoTemp-o3 明确指出仅用 IoU 作为定位奖励会诱发 reward hacking（模型可能任意定位以套利），因此提出 penalty-aware IoU reward 以惩罚低质量定位行为并鼓励稳定的多轮细化 [4]；VITAL 同样观察到多任务 RL中存在 task-wise 与 sample-wise 的难度不均衡，并通过 difficulty-aware 的方法缓解训练偏置，同时用格式奖励稳定工具调用 [2]；MedScope 将工具证据对齐纳入复合奖励（对 crop_video 用 temporaloverlap/IoU，对 get_frame 用时间戳对齐），并与格式检查共同约束策略空间 [6]。这些工作共同说明：工具调用并不天然等价于“获取更好证据”，而需要通过奖励与约束把“调用是否正确”“证据是否对齐”“是否存在投机路径”显式化。与此同时，奖励的可计算性往往迫使研究者选择易度量但可能偏狭的代理指标（如 IoU、时间戳误差），这又会反过来塑造模型偏好并带来新的泛化缺口。

面向未来的工具方向，现有证据表明最值得系统化研究的并非单一算子的“更强版本”，而是围绕证据生命周期的工具化基础设施：其一是更强的“信息路由与预算调度”工具，把 GLIMPSE 所揭示的“冗余导致性能下降”现象转化为可学习的选择机制，并与像 ModaRoute 这样的模态路由结合，实现跨模态、跨时长的证据分配 [12, 7]。其二是“可验证证据表示”的标准化，把 Video-VER 提出的“问题相关视觉证据”从训练期奖励信号推广为可共享的中间表示，使工具链不仅返回片段/帧，还返回结构化观察与可检查的支持理由，从而缓解 VideoExplorer 所面临的 verifier 误判与证据遗漏风险 [3, 10]。其三是“工具链失效诊断与反事实验证”：在 VITAL 已观察到某些工具会显著降低mIoU 的情况下，仅靠平均性能难以解释工具为何伤害，迫切需要能定位是“路由错误、验证误判、采样冗余还是索引偏差”导致的失败剖析框架 [2]。最后，数据与复现层面需要更透明的轨迹生成与筛选协议：当多项工作都依赖 Gemini-2.5-Pro、GPT-4.1 等外部模型生成或路由时，若缺少可替代的开源教师与标准化提示，工具使用研究将难以形成可累积的比较基线与可验证的结论 [4, 5, 7]。
