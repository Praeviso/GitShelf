# 5 工具谱系的分类维度与对比框架

围绕“think with video”的工具使用（tool using），若只用“是否能放大（zoom）、是否能拆帧/抽帧”来划分工具，往往会把大量关键差异（例如：工具是否显式对齐证据、是否支持反复细化定位、是否依赖离线索引、是否通过路由策略决定调用对象等）压缩为同一种“多看几眼”的直觉。更可比的做法是：用一组稳定的分类维度描述不同论文/系统的工具箱，并在每个维度下记录可观察要素——工具输入/输出是否可执行、与推理链耦合在何处、是否支持多轮调用与粒度调整、是否存在证据可靠性闭环等——从而将差异落实到可对照的接口与流程层。

首先可用“作用轴”来区分工具主要在时间维度还是空间维度上改变可见证据。时间维度工具的共同点是把连续视频压缩为可被精确复访的时间片段，并允许在时间轴上做“搜索—重访—再搜索”的迭代，而不仅是一次性均匀采样。VideoExplorer 将这一思路明确为“动态 temporal grounding $^ +$ temporal-scalable perception”的统一范式：规划器分解子问题，时间定位器把抽象信息需求变成对候选片段的检索、验证、总结等动作，并把通过验证的时间片段返还给规划器；随后感知模块再根据任务要求在该区间内做密采样或粗采样，实现时间粒度的动态调整 [3]。Video-CoM 则把时间维度工具以原子操作形式显式化：除空间放大（zoom）外，提供 find-segment（按段回看短 clip）与find-frame（定位单帧），并将多步操作组织为一条可解释的交互轨迹 $\tau = \{ ( T _ { i } , V _ { i } , m _ { i } ) \} _ { i = 1 } ^ { N }$ ；这里“工

具”不仅是取证动作本身，也包括把索引（segment/frame ID）叠加到画面上以便精确调用的工程化设计 [5]。与之相比，VideoTemp-o3 以“localize-crop-answer”概括工具链：先定位时间区间，再裁剪该区间并在裁剪后执行更密的采样取证，且允许多轮 refinement 直到证据充分 [4]。这些工作共同表明，时间轴工具的关键增量不在“抽几帧”，而在“把时间作为可操作对象”，让定位、裁剪、密采样与再定位构成可循环的取证过程。

空间维度工具对应“在同一时间片段内改变空间可见性”，典型代表是局部裁剪/放大（spatial-zoom）以读取文字、辨别细节等 [5]。但在综述口径上，空间维度不应只等同于 zoom：更一般地，它包括任何能在保持时间上下文的同时改变空间分辨率、关注区域或可见证据范围的操作。例如VideoTemp-o3 明确把 cropping 作为核心工具调用之一，并强调其与时间定位/多轮细化的配套关系：cropping 并非单次前处理，而是“按需裁剪 $^ +$ 反复细化”闭环的一部分 [4]。因此在对比时，可将“空间工具”表述为：是否支持输出可执行的空间操作参数（如裁剪框/区域 id）、是否与时间定位耦合（先定位再裁剪还是先裁剪再定位）、以及是否能在多轮交互中不断缩小空间证据范围。

第二个维度是“证据获取工具 vs 证据核验工具”。证据获取（acquisition）关注从长视频中提出候选证据并将其转化为模型可消费的上下文，例如检索片段、裁剪片段、在片段内密采样等；证据核验（verification）则回答“取回来的是否真相关、是否足以支撑当前推理步骤”。VideoExplorer 在temporal grounder 内部显式加入 verification：使用 VLM 作为 verifier 评估检索片段相关性，无关结果丢弃、相关片段返还，从而降低无关片段诱发的幻觉并让高层规划聚焦于任务拆解 [3]。VideoTemp-o3 的数据构建流程也把核验机制制度化：在生成候选区间并裁剪后进行闭环一致性验证（closed-loopconsistency verification），仅保留最终答案与 ground-truth 一致的样本；对长视频（ $> 3 \mathrm { { m i n } }$ ）还可多一轮 refinement，通过“再定位—再验证”的方式形成 multi-tool-call 训练数据 [4]。当对比不同系统的工具箱时，若只统计“有哪些操作”，容易忽略“是否有核验闭环”这一决定可靠性的结构差异；因此建议把核验工具作为与获取工具并列的一等类工具，并记录其触发条件（每次检索后都验证还是只在不确定时验证）、核验模型/规则的独立性（是否与主推理模型解耦）以及核验失败后的回退策略（丢弃、重检索或扩大窗口）。

第三个维度是“离线索引型工具 vs 在线交互型工具”。离线索引型工具强调对视频进行预处理与结构化存储，使在线推理时可以低成本地访问候选证据；在线交互型工具则强调在推理过程中即时产生操作参数、即时执行并把结果回填给推理链。VideoExplorer 明确采用离线分段与多模态嵌入库存储：离线阶段将视频切为 clips 并存为多模态 embedding；在线阶段 grounding agent以文本或多模态查询匹配候选片段，并将“construction/retrieval/verification/summarization”动作结果返回给 planner [3]。与此不同，Video-CoM 更接近在线交互式工作台：模型在每一步选择find-segment/find-frame/spatial-zoom 等操作，直接从原视频中截取片段/帧或裁剪区域并继续推理[5]。在检索侧，ModaRoute 提供了另一种“离线索引 $^ +$ 在线路由”的对比样例：系统预先构建ASR/OCR/视觉三类索引，而在线由 LLM 路由器分析查询意图后只选择部分模态索引进行检索，平均每次查询使用 1.78 个模态而非穷举 3.0 个模态，并报告约 $4 1 \%$ 的计算开销降低，同时维持Recall $\textcircled { \mathrm { { a } } 5 { = } 6 0 . 9 \% }$ [7]。这提示在视频推理工具谱系中，“索引是否离线、在线调用是否选择性”是效率与可扩展性的重要分野。

第四个维度是“单次调用 vs 多轮细化”，用于刻画工具是否支持渐进式收敛到证据。多轮细化并不只是“多调用几次”，而是要求系统状态可携带（上一轮的证据、置信度、失败原因）并能驱动下一轮缩小搜索空间或改变采样预算。VideoTemp-o3 将 multi-turn grounding refinement 作为显式能力，并在数据层区分 multi-turn single-tool-call 与 multi-turn multi-tool-call：先生成候选片段并裁剪，若一致性验证失败则允许额外 refinement；同时作者把该流程限制在长视频（ $\mathrm { > } 3 \mathrm { m i n }$ ），强调多轮细化与长时证据稀疏性之间的对应关系 [4]。VideoExplorer 的迭代机制同样体现“提出子问题—定位片段—调整粒度—再提出子问题”的循环，并在 token 效率分析中强调其通过聚焦高相关片

段实现“更少视觉 token 更好性能”[3]。在比较时，可把“是否多轮”进一步细分为：是否支持（1）时间边界的收缩/扩张，（2）采样密度的升降，（3）核验失败后的策略性回退，以及（4）多轮轨迹是否被用于训练（例如只监督末端回合还是监督全过程）。

第五个维度是“硬工具（可执行操作）vs 软工具（路由/策略/提示）”。硬工具是直接改变可见输入的操作（裁剪、检索片段、冻结帧、空间 zoom 等），其输出通常是可执行参数（时间段、帧号、裁剪框）与新的视觉上下文；软工具则不直接改动视频，而是决定“何时用什么工具、用哪个索引、以何种粒度与预算去看”，常体现为路由器、规划器、格式约束或训练期奖励等。VideoExplorer 的planner 就属于高层软工具：它不直接“切视频”，但通过分解任务、提出子问题并指定所需时间区间与粒度，扮演调度器/控制器角色 [3]。ModaRoute 的 LLM router 也是软工具：它根据查询语言结构与语义预测哪些模态最可能包含信息，并进行 selective querying；系统还把“模态选择 $^ +$ 查询优化”合并在同一提示中，从而把检索调用策略工具化、程序化 [7]。在训练对齐层面，VITAL 通过强制 rollout 输出具备 <tool call>{...}</tool call> 的格式，并施加严格的 rule-based 格式奖励 $R _ { f o r m a t } ( \tau )$ ，把“工具调用以结构化接口输出”本身作为一种稳定化机制 [2]；Video-CoM 则进一步用逐步推理奖励对每一次 manipulation 的正确性打分（ $\begin{array} { r } { \left( R _ { r e a s o n } = ( 1 / N ) \sum _ { i } C _ { i } \right) } \end{array}$ ），把“工具是否用对”从最终答案中解耦出来 [5]。Video-VER 走的是另一条“软工具”路线：它在训练期使用外部多模态大模型生成与问题相关的视觉证据，并用 Visual Evidence Reward 显式奖励“可验证地扎根于视觉证据”的推理轨迹；同时强调外部视觉语言模型仅用于训练数据与奖励构建，测试期由策略模型独立推理，从而在“离线教师—在线推理主体”之间划分角色 [10]。这些例子表明，软工具往往决定系统是否能在预算受限、证据稀疏与易幻觉的条件下稳定运作；因此在对比框架中应把软工具与硬工具并列记录，而不是把路由/规划/奖励当作“非工具的实现细节”。

将上述维度组合起来，可以形成较统一的对比口径：对任一 think-with-video 系统，先说明其工具箱覆盖的“时/空操作轴”，再说明其在“证据获取—证据核验”链路上分别有哪些组件、是否解耦；随后交代其依赖的离线索引/预处理与在线交互接口；最后用“多轮细化能力”与“硬/软工具分工”刻画其可扩展性与可靠性来源。按此口径，诸如“find-segment（重访时间片段）”“videoclipping/cropping（裁剪生成可操作的证据上下文）”“dense sampling（按需提高采样密度）”“verifier（过滤无关片段）”“closed-loop consistency verification（闭环一致性核验）”“LLM routing（选择性检索哪些模态索引）”以及“格式奖励/逐步奖励/证据奖励（将工具正确性与证据对齐显式化）”等机制，会自然落入不同维度的对比坐标，而不再被简化为放大与拆帧的同义替换 [2, 3, 4, 5, 7, 10]。
